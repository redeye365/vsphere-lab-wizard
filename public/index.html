What are you running this on?

The physical host's specs decide how big the nested cluster can realistically be.

Your physical hardware defines your constraints. Every design decision we make has to work within what you actually have. There are no wrong answers here — but being honest about your constraints leads to a better design.

Each physical host can run several nested ESXi VMs. If you have more than one, the wizard will ask for each host's IP and can distribute nested VMs across them.

This wizard is optimised for home lab deployments of 1–3 physical hosts. More than 3 is supported but untested — proceed with caution.

Which ESXi version are you deploying?

Sets the GuestId in the generated scripts and triggers version-specific minimum resource checks.

ESXi 8.x and 9.x all require at least 8GB vRAM per nested host. The wizard warns if the nested host sizing falls below the minimum. ESXi 9.x requires a VCF/VVF subscription to download.

CPU compatibility: ESXi 9.0+ dropped support for Intel Skylake, Kaby Lake-S, Broadwell-EP/DE, and 1st-Gen Xeon Scalable (Platinum 8100, Gold 6100/5100, Silver 4100, Bronze 3100). These require allowLegacyCPU=true to install — the wizard enables it in the nested cluster step by default.
AMD Ryzen (Zen 4/5): known high CPU utilisation at boot due to slow entropy generation — a workaround command is included in the generated build guide. NVMe memory tiering now works inside nested VMs in ESXi 9.1 using sched.mem.enableNestedTiering = true on each nested host.
ESXi 9.x requires an active VVF or VCF subscription to download from Broadcom. ESXi 8.x is available for free with a Broadcom account.

Top-of-rack virtual router?

A VyOS VM acts as the lab's router and firewall — gives nested hosts a routable path and provides NAT, DHCP, and DNS forwarding.

Your virtual router choice shapes your networking options throughout the lab. Consider the trade-offs before deciding:

VyOS
Open source, highly configurable, supports BGP — steeper learning curve but teaches real routing concepts used in enterprise VMware environments.
Best for certification study
pfSense
GUI-driven, easier to start with, good firewall features — less relevant to enterprise VMware environments.
Best for general networking
No router
Simplest setup — limits what you can learn about network segmentation. Fine for basic vSphere practice.
Limits learning scope

VyOS is free and open source. The script builds the VM shell and attaches the ISO; you configure it manually after boot. Very low overhead — 2 vCPU, 1GB RAM.

BGP adds the peering config to the markdown walkthrough. Both modes produce the same VM shell — it's the post-install config that differs.

What does the network around this look like?

Not the lab itself yet — the network the physical host already sits on.

Understanding your existing network is critical — your lab has to coexist with it. The most common lab networking mistakes come from not thinking through how lab traffic interacts with the home or office network. An IP conflict or broadcast storm from the lab can knock out your main network.

If no, the nested networks below will need to share a VLAN or run untagged.

Lay out the lab's networks

Each one becomes its own tagged port group on the physical vSwitch.

Network segmentation is one of the most important design decisions in any VMware environment. We are going to design separate VLANs for different traffic types — here is why each one matters:

ManagementESXi host management, vCenter, and infrastructure services. Needs to be reachable but should be separated from VM workload traffic.
vMotionLive migration traffic between hosts. High bandwidth, latency-sensitive. Should never share a VLAN with management or VM traffic.
vSANStorage replication between hosts. Very high bandwidth. Dedicated VLAN prevents storage I/O from affecting management or vMotion.
TEP (NSX)Tunnel Endpoint traffic for NSX overlay networking. Required if you are deploying NSX — carries all east-west VM traffic over VXLAN/Geneve tunnels.
VM networkWorkload VM traffic. Separate from infrastructure traffic so a misbehaving VM can't affect host management.
External port group

Has a physical uplink to your home or office network and the internet. Your lab reaches the outside world through this. In a standard ESXi install this is called VM Network. VyOS connects its WAN interface here.

Internal lab port group

No physical uplink — traffic stays entirely inside the physical host. All inter-VLAN routing is handled by VyOS. Set this port group to VLAN 4095 (trunk mode) so VyOS receives tagged frames for every VLAN.

Management traffic runs on native VLAN. Port group, VyOS, and nested vmk0 all untagged.

All three layers must use the same VLAN ID: physical port group, VyOS eth1 vif N, and nested vmk0.

vSAN network appears here because it’s switched on in the next step.

Cluster design involves balancing three things that often conflict: availability (more hosts = more resilience), resource headroom (more VMs need more capacity), and cost (more hosts = more hardware). There is no universally right answer — it depends on your use case.

Resource saving tips

Add NSX-T to this lab?

NSX-T adds software-defined networking, micro-segmentation, and optional BGP routing on top of the vSphere cluster. Deployment is after the cluster is healthy.

NSX adds significant capability — micro-segmentation, overlay networking, distributed routing — but also significant complexity and resource overhead. Before choosing NSX, be clear on why you need it.

✓ Learning NSX for certification (VCP-VCF Admin, VCP-VCF Architect, VCAP Networking)
✓ Practising micro-segmentation for a customer project
✓ Building a full VCF environment

NSX Manager deploys as a VM on the cluster. Small size works for most labs.

For VCF: NSX must be configured to Broadcom’s BOM sizing and topology requirements. This lab topology is suitable for learning and practice, not for a VCF management domain bring-up.

Edge cluster is created by the configure script after you deploy the edge node(s) from the NSX UI.

Must be in the management network range. Used in generated scripts.

NSX T0 (AS 65001) will peer with VyOS (AS 65002). These are auto-populated from the VyOS BGP selection. Adjust if you use different ASNs.

These CIDRs are added as a BGP outbound prefix list on the T0 gateway and applied to the VyOS neighbour.

VCF bundle depot?

A local HTTPS file server hosting VCF bundles on your LAN. During VCF bring-up, SDDC Manager downloads bundles from Broadcom's CDN — a local depot gives LAN-speed downloads instead and enables air-gapped deployments.

This sets up the depot server infrastructure only. Downloading the actual VCF bundles from Broadcom's portal and populating the depot is a manual step covered in depot-instructions.md. VCF bring-up itself is out of scope for v1 of this wizard.

Small Ubuntu VM: 2 vCPU / 4GB RAM / 100GB disk on host 1's local datastore. nginx serves bundles over HTTPS with a self-signed cert. Generates depot-deploy.ps1 and depot-configure.sh.

Installs IIS on the existing domain controller and configures HTTPS using a Windows self-signed cert. No extra VM needed. Generates depot-iis.ps1.

IIS mode requires a domain controller — enable the DC in step 5 or switch to Linux/nginx mode.

Must be in the management network range. Used in the depot URL: https://<ip>/depot/
