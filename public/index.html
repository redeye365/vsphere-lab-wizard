Which ESXi version are you deploying?

Sets the GuestId in the generated scripts and triggers version-specific minimum resource checks.

ESXi 8.0 needs at least 8GB vRAM per nested host. ESXi 7.0 and 6.7 need at least 4GB. The wizard will warn you if the nested host sizing falls below the minimum for the version you pick.

Top-of-rack virtual router?

A VyOS VM acts as the lab's router and firewall — gives nested hosts a routable path and provides NAT, DHCP, and DNS forwarding.

VyOS is free and open source. The script builds the VM shell and attaches the ISO; you configure it manually after boot. Very low overhead — 2 vCPU, 1GB RAM.

BGP adds the peering config to the markdown walkthrough. Both modes produce the same VM shell — it's the post-install config that differs.

Domain controller?

A Windows DC gives the lab DNS, NTP, and Active Directory — required before vCenter is deployed if you want valid FQDNs and certificates.

Sized at 2 vCPU / 4GB RAM / 80GB disk. Windows Server ISO install is manual — same pattern as the nested ESXi hosts.

e.g. lab.acme.internal — used in the DC promotion command and the vCenter FQDN.

This is checked against the lab network ranges in the next steps to flag IP conflicts.

Nested host disk layout

Virtual disks added to each nested ESXi host VM, on top of the boot disk. Configure vSAN capacity/cache and the local datastore for vCenter bootstrap here.

Add one entry per disk role. vSAN capacity and cache disks are added to every nested host; the local datastore disk goes to host 1 only. Leave blank if your nested hosts only need a boot disk.

Deploy test workload VMs?

Blank VM shells on the cluster for practising vMotion, DRS, and HA — no OS required, but you can install one later.

Generates a deploy-workloads.ps1 script that runs after vSAN is healthy. VMs have no OS but are useful for DRS/HA/vMotion practice.
